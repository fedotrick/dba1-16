## Модуль 03: Конфигурирование сервера

### Параметры конфигурации
- Управляют работой и поведением СУБД
- Уровни установки:
  1. **Экземпляр** — файлы конфигурации
  2. **База данных / пользователь** — через ALTER DATABASE / ALTER ROLE
  3. **Сеанс** — через SET

### Файлы конфигурации

| Файл | Назначение |
|------|-----------|
| **postgresql.conf** | Основной файл параметров (ключ=значение) |
| **pg_hba.conf** | Управление подключением (аутентификация) |
| **pg_ident.conf** | Отображение имён пользователей ОС на роли БД |

### postgresql.conf
- Считывается при старте сервера
- По умолчанию: в PGDATA, но пакетные дистрибутивы размещают иначе
  - Ubuntu: `/etc/postgresql/16/main/`
- Можно подключать доп. файлы: `include`, `include_dir`, `include_if_exists`
- Если параметр указан несколько раз — используется **последнее** значение
- Узнать расположение: `SHOW config_file;`

### Применение изменений
| Команда | Описание |
|---------|----------|
| `pg_ctl reload` | Перечитать конфигурацию (без перезапуска) |
| `pg_ctlcluster 16 main reload` | То же в Ubuntu |
| `SELECT pg_reload_conf();` | То же из SQL |
| Перезапуск сервера | Требуется для параметров с context = 'postmaster' |

### Представление pg_file_settings
Показывает все параметры из файлов конфигурации:
- `sourcefile`, `sourceline` — откуда прочитан
- `applied` — применён ли
- `error` — ошибка, если есть

### Контексты параметров (pg_settings.context)
| Контекст | Как применить |
|----------|--------------|
| **postmaster** | Требуется перезапуск сервера |
| **sighup** | Достаточно reload (перечитать) |
| **user** | Можно установить через SET (уровень сеанса) |
| **superuser** | Можно установить через SET только суперпользователем |
| **backend** | Только при запуске сервера |

### ALTER SYSTEM — изменение параметров
```sql
ALTER SYSTEM SET work_mem = '64MB';
ALTER SYSTEM RESET work_mem;
ALTER SYSTEM RESET ALL;
```
Результат записывается в **postgresql.auto.conf** (в PGDATA).
Этот файл считывается **после** postgresql.conf, поэтому его параметры имеют приоритет.

### Уровни установки параметров (приоритет от высшего к низшему)
1. Командная строка запуска (`postgres -c ...`)
2. `postgresql.auto.conf` (ALTER SYSTEM)
3. `postgresql.conf` и подключённые файлы (включая `include_dir`)
4. `ALTER DATABASE ... SET` — для сеансов в конкретной БД
5. `ALTER ROLE ... SET` — для сеансов конкретного пользователя/роли
6. `SET` — для текущего сеанса

### Параметры в командной строке приложения
```bash
# Через options в строке подключения
psql "options='-c work_mem=32MB'"

# Через переменную окружения
export PGOPTIONS='-c work_mem=32MB'
```

### Узнать командную строку запуска сервера
```bash
# PID из postmaster.pid
cat $PGDATA/postmaster.pid | head -n 1
# Затем
ps -p <PID> -o command
```

### Представление pg_settings
```sql
-- Все параметры
SELECT * FROM pg_settings;

-- Параметры, требующие перезапуска
SELECT name, setting, unit FROM pg_settings WHERE context = 'postmaster';

-- Конкретный параметр
SELECT * FROM pg_settings WHERE name = 'work_mem';
```

## Модуль 04: Общее устройство PostgreSQL

### Клиент-серверное взаимодействие
- Клиент подключается → аутентификация → запросы → результат
- Драйверы: libpq (C), psycopg2 (Python), JDBC (Java) и др.
- Протокол: https://postgrespro.ru/docs/postgresql/16/protocol

### ACID-свойства транзакций
| Свойство | Описание |
|----------|----------|
| **Атомарность** | Всё или ничего (BEGIN, COMMIT, ROLLBACK) |
| **Согласованность** | Переводит БД из одного согласованного состояния в другое |
| **Изоляция** | Параллельные транзакции не влияют друг на друга |
| **Долговечность** | Зафиксированные данные не теряются при сбое |

### Точки сохранения (SAVEPOINT)
```sql
BEGIN;
SAVEPOINT sp;
-- операции
ROLLBACK TO sp;  -- откат к точке сохранения (не отменяет транзакцию!)
COMMIT;
```

### Путь выполнения запроса
1. **Разбор** (parser) — синтаксический и семантический анализ → дерево разбора
2. **Переписывание** (rewriter) — правила, подстановка представлений
3. **Планирование** (planner) — план выполнения на основе статистики
4. **Выполнение** (executor) — выполнение плана → результат

Для семантического анализа нужен **системный каталог**.
Для планирования — **статистика** по данным.

### Подготовленные операторы (PREPARE/EXECUTE)
```sql
PREPARE q(integer) AS SELECT * FROM t WHERE id = $1;
EXECUTE q(1);
```
- Разбор и переписывание выполняются **один раз** при PREPARE
- Защита от SQL-инъекций
- Просмотр: `SELECT * FROM pg_prepared_statements;`

### Курсоры
```sql
BEGIN;
DECLARE c CURSOR FOR SELECT * FROM t;
FETCH 10 c;  -- получить 10 строк
CLOSE c;
COMMIT;
```
- Курсор = портал на сервере
- Автоматически закрываются по завершении транзакции
- `WITH HOLD` — курсор сохраняется после COMMIT

### Процессы PostgreSQL
| Процесс | Назначение |
|---------|-----------|
| **postmaster** | Главный процесс, управляет всеми остальными, слушает соединения |
| **backend** | Обслуживающий процесс для каждого клиента (fork от postmaster) |
| **bgwriter** | Фоновая запись «грязных» буферов (предвосхищение) |
| **checkpointer** | Сброс всех грязных буферов (контрольная точка) |
| **walwriter** | Асинхронная запись WAL |
| **autovacuum launcher** | Запуск процессов автоочистки |

### Память
| Тип | Назначение |
|-----|-----------|
| **Общая память** | Буферный кеш, WAL-буферы, разделяемые структуры |
| **Локальная память backend** | Разобранные запросы, курсоры, кеш каталога, sort/join (work_mem) |

### Хранение данных
- Страница = **8 КБ** (можно 16 или 32 КБ только при сборке)
- Страница: заголовок + данные + свободное пространство
- **Буферный кеш** в общей памяти — все процессы имеют доступ
- ОС тоже кеширует данные (дополнительный кеш)
- **WAL** защищает данные от потери при сбое (долговечность)

### Пул соединений
- **pgBouncer** — менеджер пула соединений
- Клиенты подключаются к pgBouncer, а не напрямую к PostgreSQL
- pgBouncer может приостанавливать клиентов (для обновления сервера)

### Расширяемость
- Новые типы данных, индексы, функции, операторы
- Языки: SQL, PL/pgSQL, PL/Perl, PL/Python, PL/Tcl
- Индексы: B-tree, Hash, GiST, SP-GiST, GIN, BRIN
- FDW (Foreign Data Wrapper) — внешние источники данных
- Фоновые рабочие процессы
- Большинство расширений устанавливаются «на лету»

## Модуль 05: Изоляция и многоверсионность (MVCC)

### Многоверсионность (MVCC)
- Несколько версий одной строки хранятся в файлах данных
- Версии различаются **временем действия** = номер транзакции (xid)
- INSERT: версия помечается xmin = номер транзакции
- DELETE: версия помечается xmax = номер транзакции (физически не удаляется!)
- UPDATE = DELETE + INSERT

### Снимок данных (Snapshot)
Согласованный срез на определённый момент времени:
- Номер последней зафиксированной транзакции
- Список активных (незавершённых) транзакций

### Уровни изоляции
| Уровень | Когда строится снимок | Поведение |
|---------|----------------------|-----------|
| **Read Uncommitted** | Не поддерживается → работает как Read Committed |
| **Read Committed** (по умолчанию) | Начало **каждого оператора** | Одинаковые запросы могут давать разные результаты |
| **Repeatable Read** | Начало **первого оператора** транзакции | Все запросы видят одни данные. Возможна ошибка сериализации |
| **Serializable** | Начало первого оператора | Полная изоляция. Возможна ошибка сериализации |

```sql
BEGIN ISOLATION LEVEL REPEATABLE READ;
```

### Очистка и горизонт очистки
- **Горизонт очистки** — номер xid, до которого все старые версии уже не видны ни в одном снимке
- Один горизонт на каждую БД
- Долгие транзакции могут удерживать горизонт, мешая очистке

### Блокировки при MVCC
- **Чтение никогда не блокирует** ни читающие, ни пишущие транзакции
- **Изменение строки** не блокирует её чтение
- Блокировка только если две транзакции пытаются **одновременно менять** одну строку
- Блокировки на таблицах: запрет удаления таблицы, пока идёт работа
- Блокировки снимаются при завершении транзакции или откате к SAVEPOINT

### Статус транзакций (clog)
- Файлы в `PGDATA/pg_xact/`
- **2 бита** на транзакцию
- Буферы в общей памяти
- **Фиксация и обрыв одинаково быстры** — просто ставятся биты
- При обрыве данные НЕ откатываются физически — просто ставится бит «прервана»

### Скрытые столбцы
- `xmin` — xid транзакции, создавшей версию строки
- `xmax` — xid транзакции, удалившей версию (0 = не удалена)
- Скрыты, использование не рекомендуется в реальной работе

## Модуль 06: Очистка (VACUUM)

### Задачи очистки
1. **Вычищение мёртвых версий строк** из таблиц и индексов
2. **Обновление карты видимости** — страницы, где все версии видны во всех снимках
3. **Обновление карты свободного пространства** — свободное место в страницах
4. **Сбор статистики** для планировщика (ANALYZE)
5. **Заморозка старых версий** — защита от переполнения счётчика транзакций

### Карта видимости
- Страницы, содержащие только актуальные версии, видимые во всех снимках
- Применяется для:
  - Оптимизации очистки (не нужно заглядывать в такие страницы)
  - Index Only Scan (можно не обращаться к таблице)
- Есть **только для таблиц**

### Карта свободного пространства
- Объём пустого места в страницах
- Используется при вставке новых строк для быстрого поиска подходящей страницы
- Есть для **таблиц и индексов**

### Автоочистка (autovacuum)
- Фоновый процесс `autovacuum launcher` запускает `autovacuum worker` для каждой БД
- Параметры:
  - `autovacuum = on/off` — вкл/выкл
  - `autovacuum_max_workers` — макс. рабочих процессов (по умолч. 3)
  - `autovacuum_naptime` — пауза между запусками (по умолч. 1 мин)
  - `autovacuum_vacuum_threshold` = 50 — мин. обновлений для VACUUM
  - `autovacuum_analyze_threshold` = 50 — мин. обновлений для ANALYZE
  - `autovacuum_vacuum_scale_factor` = 0.2 — доля от размера таблицы
  - `autovacuum_vacuum_cost_delay` = 2ms — задержка для снижения нагрузки

### VACUUM FULL и REINDEX
| Команда | Что делает | Блокировка |
|---------|-----------|-----------|
| `VACUUM FULL` | Полная перезапись таблиц и индексов, возврат места ОС | **EXCLUSIVE** — полная блокировка |
| `REINDEX` | Перестроение индексов | Блокирует запись и использование индекса |
| `REINDEX CONCURRENTLY` | Перестроение без блокировки таблицы | Не блокирует запись |

> **Важно:** REINDEX CONCURRENTLY — не транзакционно, дольше, может завершиться неудачей. Не работает для системных индексов и индексов EXCLUDE. Не может выполняться внутри транзакции.

### Проблема разрастания (bloat)
- VACUUM **не уменьшает** размер файлов — место не возвращается ОС
- Причины разрастания: неправильная настройка автоочистки, массовое изменение, долгие транзакции
- Индексы (B-деревья): страницы расщепляются, но никогда не объединяются

### Оценка разрастания
- `pg_table_size()`, `pg_indexes_size()`, `pg_total_relation_size()`
- `pg_size_pretty()` — форматирование
- Расширение `pgstattuple`:
  - `pgstattuple()` — точная статистика таблицы
  - `pgstattuple_approx()` — приблизительная (быстрее)
  - `pgstatindex()` — статистика индекса

## Модуль 07: Буферный кеш и журнал (WAL)

### Буферный кеш
- Массив буферов в **общей памяти**
- Размер страницы: 8 КБ (по умолчанию)
- **LRU** (Least Recently Used) — алгоритм вытеснения
- Параметр: `shared_buffers` (по умолчанию 128 MB — мало для продакшена)
- «Грязный» буфер — изменённый, подлежит записи на диск

```sql
-- Проверить попадание в кеш
EXPLAIN (ANALYZE, BUFFERS) SELECT * FROM t;
-- hit = из кеша, read = с диска
```

### Журнал предзаписи (WAL — Write-Ahead Logging)
- Поток записей о выполняемых действиях
- Запись попадает на диск **раньше**, чем изменённые данные
- Защищает: страницы таблиц, индексов, clog
- **НЕ защищает:** временные и нежурналируемые (UNLOGGED) таблицы
- Файлы в `PGDATA/pg_wal/`, размер по умолчанию **16 МБ**

### LSN (Log Sequence Number)
- 64-разрядное число — смещение в байтах от начала журнала
- Записывается как два 32-битных числа: `0/2378D28`
- `pg_current_wal_lsn()` — текущая позиция
- `pg_ls_waldir()` — файлы журнала

### Контрольная точка (checkpoint)
- Периодический сброс **всех** грязных буферов на диск
- Гарантирует, что все изменения до КТ — на диске
- Ограничивает размер журнала для восстановления
- Восстановление после сбоя: с последней КТ + проигрывание WAL

### Процессы кеша и журнала
| Процесс | Назначение |
|---------|-----------|
| **walwriter** | Асинхронная запись WAL на диск |
| **checkpointer** | Сброс всех грязных буферов (контрольная точка) |
| **bgwriter** | Запись части грязных буферов (предвосхищение вытеснения) |
| **backend** | Чтение данных, самостоятельная запись грязного буфера при вытеснении |

### Режимы записи WAL
- **Синхронный** — fsync при фиксации транзакции (надёжно)
- **Асинхронный** — walwriter пишет постепенно с задержкой (быстрее)
- Оба режима работают **совместно**

### Уровни WAL (wal_level)
| Уровень | Назначение |
|---------|-----------|
| **minimal** | Только восстановление после сбоя |
| **replica** (по умолчанию) | + бэкапы, физическая репликация |
| **logical** | + логическая репликация |

### Режимы останова сервера
| Режим | Команда | Поведение |
|-------|---------|-----------|
| **smart** | `pg_ctl stop -m smart` | Ждёт отключения всех клиентов |
| **fast** (по умолч.) | `pg_ctl stop -m fast` | Откат транзакций, checkpoint, останов |
| **immediate** | `pg_ctl stop -m immediate` | Мгновенно, **без checkpoint** → при запуске восстановление через WAL |

## Модуль 08: Базы данных и схемы

### Кластер баз данных
При инициализации создаются 3 БД:
| БД | Назначение |
|----|-----------|
| **template1** | Шаблон по умолчанию. Можно добавлять объекты → попадут в каждую новую БД |
| **template0** | Не изменять! Для восстановления из pg_dump и создания БД с другой кодировкой |
| **postgres** | БД по умолчанию для пользователя postgres |

### Таблица pg_database
| Столбец | Описание |
|---------|----------|
| `datname` | Имя БД |
| `datistemplate` | Является ли шаблоном |
| `datallowconn` | Разрешены ли соединения |
| `datconnlimit` | Макс. кол-во соединений (-1 = без ограничений) |

### Создание БД
```sql
CREATE DATABASE db;
CREATE DATABASE db TEMPLATE template0 ENCODING 'UTF8';
ALTER DATABASE db RENAME TO appdb;
ALTER DATABASE appdb CONNECTION LIMIT 10;
SELECT pg_size_pretty(pg_database_size('appdb'));
DROP DATABASE appdb;  -- нельзя удалить, если есть подключения!
```

### Схемы
- Пространство имён для объектов внутри БД
- **Схема ≠ пользователь**
```sql
CREATE SCHEMA app;
ALTER TABLE t SET SCHEMA app;
DROP SCHEMA app CASCADE;
```

### Путь поиска (search_path)
```sql
SHOW search_path;  -- по умолчанию: "$user", public
SET search_path = public, app;
ALTER DATABASE appdb SET search_path = public, app;
SELECT current_schema();      -- первая обычная схема (для CREATE)
SELECT current_schemas(true); -- все схемы, включая неявные
```

### Специальные схемы
| Схема | Описание |
|-------|----------|
| **public** | По умолчанию в search_path |
| **$user** | Не существует по умолчанию. Если создать — объекты пользователя будут там |
| **pg_catalog** | Системный каталог. Если нет в пути — подразумевается **первой** |
| **pg_temp** | Временная схема. Подразумевается **самой первой**, даже если нет в пути |

### Временные таблицы
- Существуют на время сеанса или транзакции
- **Не журналируются** (нет восстановления после сбоя)
- **Не попадают** в общий буферный кеш
- Параметр: `temp_buffers`
```sql
CREATE TEMP TABLE t (s text);
SELECT * FROM pg_temp.t;
```

### Настройки на уровне БД и роли
```sql
ALTER DATABASE appdb SET search_path = "$user", app, public;
ALTER DATABASE data_databases SET temp_buffers = '32MB';
ALTER ROLE student SET work_mem = '64MB';
```
Хранятся в `pg_db_role_setting`. Просмотр: `\drds`.

## Модуль 09: Системный каталог

### Что такое системный каталог
- Набор таблиц и представлений, описывающих все объекты кластера
- Основная схема: **pg_catalog**
- Стандартное представление: **information_schema** (65 представлений в PG16)
- С PG14 — первичные ключи для большинства таблиц каталога

### Общие объекты кластера
- Таблицы, общие для всего кластера (базы данных, роли, табличные пространства)
- Хранятся вне какой-либо конкретной БД, видны из каждой

### Правила именования
- Все таблицы/представления каталога начинаются с **`pg_`**
- Столбцы имеют трёхбуквенный префикс: `pg_database.datname` (без `_` после префикса!)
- Имена объектов хранятся в **нижнем регистре**

### Ключевые таблицы системного каталога
| Таблица | Описание |
|---------|----------|

## Модуль 10: Табличные пространства

### Табличное пространство (tablespace)
- Связь между логическими объектами и файловой системой
- Позволяет разместить данные на разных дисках/носителях

### Системные табличные пространства
| ТП | Назначение |
|----|-----------|
| **pg_default** | По умолчанию для таблиц и индексов (`$PGDATA/base/`) |
| **pg_global** | Для общих объектов кластера (`$PGDATA/global/`) |

### Создание и управление
```sql
CREATE TABLESPACE fastspace LOCATION '/mnt/fastdisk';
ALTER TABLESPACE fastspace RENAME TO fast;
ALTER TABLESPACE fastspace OWNER TO user1;
DROP TABLESPACE fastspace;  -- должно быть пустым!
```

### Расположение в ФС
- В указанном LOCATION создаётся каталог с именем `PG_16_YYYYMMDD1`
- Внутри — подкаталоги с OID баз данных
- Файлы данных с именами = OID объектов

### Параметры
| Параметр | Описание |
|----------|----------|
| `default_tablespace` | ТП по умолчанию для новых объектов |
| `temp_tablespaces` | ТП для временных объектов и сортировок (можно несколько через запятую) |

### Перемещение объектов
```sql
ALTER TABLE t SET TABLESPACE fastspace;
ALTER INDEX idx SET TABLESPACE fastspace;
-- Перемещает все таблицы и индексы из одного ТП в другое:
ALTER TABLE ALL IN TABLESPACE pg_default SET TABLESPACE fastspace;
```

## Модуль 11: Низкий уровень хранения данных

### Файлы данных
- Каждая таблица/индекс хранится в одном или нескольких файлах
- Файлы сегментируются по **1 ГБ** (параметр `segment_size`)
- Именование файлов: **OID объекта**
- Узнать путь: `pg_relation_filepath(имя)`

### Слои файлов (суффиксы)
| Файл | Описание |
|------|----------|
| **основной** (без суффикса) | Основные данные |
| **_fsm** (Free Space Map) | Карта свободного пространства |
| **_vm** (Visibility Map) | Карта видимости |
| **_init** | Начальное содержимое (для unlogged таблиц) |

### Структура каталога
```
$PGDATA/base/<db_oid>/<relation_oid>      — данные
$PGDATA/base/<db_oid>/<relation_oid>_fsm  — карта свободного пространства
$PGDATA/base/<db_oid>/<relation_oid>_vm   — карта видимости
```

### Функции для определения размера
| Функция | Описание |
|---------|----------|
| `pg_relation_size(имя)` | Размер основного файла |
| `pg_relation_size(имя, 'fsm')` | Размер FSM |
| `pg_relation_filepath(имя)` | Путь к файлу отношения |
| `pg_table_size(имя)` | Таблица + FSM + VM + TOAST (без доп. индексов) |
| `pg_indexes_size(имя)` | Размер всех индексов таблицы |
| `pg_total_relation_size(имя)` | Полный размер: данные + все индексы + TOAST |
| `pg_size_pretty()` | Форматирование размера (1234 kB → «1234 kB») |

### Длинные версии строк и TOAST
**TOAST** (The Oversized-Attribute Storage Technique):
- Страница = 8 КБ, строка не помещается → TOAST
- Значение разбивается на «куски» (chunks) ~2 КБ
- Хранятся в отдельной TOAST-таблице: `pg_toast.pg_toast_<OID>`

### Стратегии хранения столбцов (attstorage)
| Стратегия | Сжатие | Вынос | Описание |
|-----------|--------|-------|----------|
| **plain** | Нет | Нет | Для фиксированных типов |
| **extended** | Да | Да | По умолчанию для可变长度的 типов. Сначала сжать, потом вынести |
| **external** | Нет | Да | Не сжимать, сразу вынести (быстрый доступ) |
| **main** | Да | Только если не влезает | Сжать, выносить только при необходимости |

```sql
-- Изменить стратегию
ALTER TABLE t ALTER COLUMN col SET STORAGE external;

-- Просмотреть
SELECT attname, attstorage FROM pg_attribute WHERE attrelid = 't'::regclass;
```

### Методы сжатия TOAST
- **pglz** (по умолчанию)
- **lz4** (быстрее, начиная с PG14)
```sql
ALTER TABLE t ALTER COLUMN col SET COMPRESSION lz4;
```

| Атрибут | Описание |

### pg_hba.conf — управление подключением
Определяет: **кто**, **откуда**, **к какой БД**, **каким методом** подключается.

Формат записи:
```
тип  БД  пользователь  адрес  метод  [опции]
```

| Тип | Описание |
|-----|----------|
| `local` | Unix-socket (локально) |

### Методы аутентификации
| Метод | Описание |
|-------|----------|

## Модуль 14: Резервное копирование

### Логическое резервное копирование

#### COPY
```sql
-- На стороне сервера (требует прав суперпользователя)
COPY t TO '/tmp/t.csv';
COPY t FROM '/tmp/t.csv';

-- На стороне клиента (через psql)
\copy t TO 't.csv';
\copy t FROM 't.csv';
```

#### pg_dump — дамп одной БД
```bash
# SQL-скрипт (plain)
pg_dump -d dbname > dump.sql

# Пользовательский формат (сжатый, можно выбирать объекты)
pg_dump -d dbname -Fc -f dump.custom

# Directory-формат (поддержка параллелизма)
pg_dump -d dbname -Fd -j 4 -f dump_dir/

# Только схема
pg_dump -d dbname -s

# Только данные
pg_dump -d dbname -a

# Конкретные таблицы/схемы
pg_dump -d dbname -t table1 -n schema1

# Без владельцев (без ALTER OWNER)
pg_dump -d dbname -O

# Без привилегий (без GRANT/REVOKE)
pg_dump -d dbname -x
```

| Формат | Ключ | Описание |
|--------|------|----------|
| plain | (по умолчанию) | Чистый SQL-скрипт |
| custom | `-Fc` | Сжатый, можно выбирать объекты при восстановлении |
| directory | `-Fd` | Каталог с файлами, поддержка параллелизма |
| tar | `-Ft` | TAR-архив |

#### pg_restore — восстановление
```bash
pg_restore -d dbname dump.custom
pg_restore -d dbname -s dump.custom    # только схема
pg_restore -d dbname -a dump.custom    # только данные
pg_restore -d dbname -j 4 dump_dir/   # параллельно
pg_restore -d dbname -c dump.custom   # удалить перед созданием
pg_restore -d dbname --clean --if-exists dump.custom
pg_restore -l dump.custom              # список объектов
```

#### pg_dumpall — дамп всего кластера
```bash
pg_dumpall > full_cluster.sql
pg_dumpall -g > roles_only.sql  # только роли и табличные пространства
```

### Физическое резервное копирование

#### pg_basebackup
```bash
# Базовый бэкап в формате plain
pg_basebackup -D /backup/base -Fp -Xs

# В формате tar со сжатием
pg_basebackup -D /backup/base -Ft -z

# С быстрым checkpoint
pg_basebackup -D /backup/base -Fp -Xs -c fast

# Распределённый checkpoint (меньше нагрузка)
pg_basebackup -D /backup/base -Fp -Xs -c spread
```

| Параметр | Описание |
|----------|----------|
| `-D` | Каталог для бэкапа |
| `-Fp`/`-Ft` | plain/tar формат |
| `-Xs`/`-Xf` | stream/fetch WAL |
| `-Z 0-9` | Уровень сжатия |
| `-c fast/spread` | Режим checkpoint |
| `-j N` | Параллельные рабочие процессы (для tar) |

### Point-in-Time Recovery (PITR)

#### Архивирование WAL
```postgresql
# postgresql.conf
wal_level = replica
archive_mode = on
archive_command = 'cp %p /archive/%f'
```
- `%p` — путь к файлу WAL
- `%f` — имя файла WAL
- Команда должна вернуть 0 **только** при успешном копировании

#### Восстановление
```bash
# 1. Остановить сервер
# 2. Очистить PGDATA
# 3. Восстановить базовый бэкап
# 4. Настроить postgresql.auto.conf:
#    restore_command = 'cp /archive/%f %p'
#    recovery_target_time = '2025-01-01 12:00:00'
# 5. Создать файл recovery.signal в PGDATA
# 6. Запустить сервер
```

#### Параметры восстановления
| Параметр | Описание |
|----------|----------|
| `restore_command` | Команда извлечения архивированных WAL |
| `recovery_target_time` | Время восстановления |
| `recovery_target_xid` | XID транзакции |
| `recovery_target_name` | Имя restore point |
| `recovery_target_lsn` | LSN |
| `recovery_target_action` | pause / shutdown / promote |
| `recovery_target_inclusive` | true/false — включать ли целевую точку |

#### Утилиты WAL
| Утилита | Описание |
|---------|----------|
| `pg_receivewal` | Непрерывный поток WAL в файлы |
| `pg_recvlogical` | Поток логического декодирования |
| `pg_waldump` | Чтение содержимого WAL-файлов |

## Модуль 15: Физическая репликация

### Механизм
- Передача и проигрывание WAL на другой сервер
- Процессы:
  - **wal sender** (на основном) — отправляет WAL
  - **wal receiver** (на реплике) — принимает WAL
  - **startup** (на реплике) — проигрывает WAL

### Требования
- Одинаковая архитектура процессора
- Одинаковая версия PostgreSQL
- На основном: `wal_level = replica`
- На основном: `max_wal_senders >= кол-во реплик`
- В pg_hba.conf: `host replication replicator <адрес> scram-sha-256`

### Настройка реплики
```bash
# pg_basebackup для начальной копии:
pg_basebackup -h primary -D /var/lib/postgresql/16/main \
    -U replicator -Xs -R -P
```
Ключ `-R` создаёт `standby.signal` и записывает `primary_conninfo` в `postgresql.auto.conf`.

### Типы репликации
| Тип | Описание |
|-----|----------|
| **Асинхронная** (по умолчанию) | Основной не ждёт подтверждения. Быстрее, возможна потеря данных |
| **Синхронная** | Основной ждёт подтверждения. Надёжнее, медленнее |

### Синхронная репликация
```postgresql
# postgresql.conf на основном
synchronous_standby_names = 'replica1'
```

### Слоты репликации
- Гарантируют, что WAL не будет удалён, пока реплика его не получит
- `max_replication_slots` — макс. кол-во слотов
- Мониторинг: `SELECT * FROM pg_replication_slots;`

### Мониторинг репликации
```sql
-- На основном
SELECT * FROM pg_stat_replication;
-- state, sent_lsn, write_lsn, flush_lsn, replay_lsn

-- На реплике
SELECT pg_is_in_recovery();            -- true = реплика
SELECT pg_last_wal_receive_lsn();      -- полученный WAL
SELECT pg_last_wal_replay_lsn();       -- применённый WAL
SELECT pg_last_xact_replay_timestamp(); -- время последнего применённого WAL
```

### Конфликты на реплике
| Параметр | Описание |
|----------|----------|
| `hot_standby_feedback = on` | Реплика сообщает основанию о своих снимках — предотвращает конфликты с vacuum |
| `max_standby_streaming_delay` | Макс. задержка транзакций на реплике (перед принудительным откатом) |

### Переключение (switchover / failover)
```sql
-- На реплике (promotion):
SELECT pg_promote();
-- Или:
pg_ctl promote
```
> **Важно:** Для предотвращения split brain — только одна реплика может стать основной.

### Каскадная репликация
- Реплика может передавать WAL другой реплике
- Снижает нагрузку на основной сервер

### Задержанная репликация
```postgresql
# postgresql.conf на реплике
recovery_min_apply_delay = '5min'
```
Полезно для защиты от случайного удаления данных.

## Модуль 16: Логическая репликация

### Архитектура
- **Издатель (publisher)** → **Подписчик (subscriber)**
- WAL декодируется в изменения строк (не в байты WAL)
- Процессы: **logical replication worker** (на подписчике), **logical launcher**

### Требования
- `wal_level = logical` на издателе
- У таблиц должен быть `REPLICA IDENTITY` (по умолчанию DEFAULT = первичный ключ)

### Публикации
```sql
-- Создание
CREATE PUBLICATION mypub FOR TABLE t1, t2;
CREATE PUBLICATION mypub FOR ALL TABLES;
CREATE PUBLICATION mypub FOR TABLE t1 (col1, col2);  -- только столбцы
CREATE PUBLICATION mypub FOR TABLE t1 WHERE (status = 'active');  -- фильтр

-- С операциями
CREATE PUBLICATION mypub FOR TABLE t1 WITH (publish = 'insert,update');

-- Изменение
ALTER PUBLICATION mypub ADD TABLE t3;
ALTER PUBLICATION mypub DROP TABLE t2;
```

### Подписки
```sql
CREATE SUBSCRIPTION mysub
    CONNECTION 'host=primary dbname=mydb user=replicator'
    PUBLICATION mypub;
```
- Начальная синхронизация данных (tablesync)
- Затем непрерывное применение изменений (apply)

### Параметры подписки
| Параметр | Описание |
|----------|----------|
| `streaming = on/off` | Потоковая передача больших транзакций до фиксации |
| `create_slot = true/false` | Создавать ли слот репликации |
| `copy_data = true/false` | Копировать ли существующие данные |
| `synchronous_commit` | Режим синхронной фиксации |
| `disable_on_error = on/off` | Отключить подписку при ошибке |

### Ограничения
| Ограничение | Описание |
|-------------|----------|
| DDL | **Не реплицируется** |
| Последовательности | Не реплицируются автоматически |
| Большие объекты | Не реплицируются |
| Представления | Не реплицируются |
| TRUNCATE | Только при `publish = 'truncate'` |

### REPLICA IDENTITY
| Значение | Описание |
|----------|----------|
| **DEFAULT** | Первичный ключ (по умолчанию) |
| **FULL** | Все столбцы |
| **USING INDEX индекс** | Указанный уникальный индекс |
| **NOTHING** | Нет идентификатора → UPDATE/DELETE не работают на подписчике |

```sql
ALTER TABLE t REPLICA IDENTITY FULL;
ALTER TABLE t REPLICA IDENTITY USING INDEX idx_name;
ALTER TABLE t REPLICA IDENTITY NOTHING;
```

### Сценарии использования
1. **Консолидация данных** из филиалов в центральный сервер
2. **Обновление без простоя** — реплика новой версии → переключение
3. **Двунаправленная репликация** (PG16+) — master-master
4. **Выборочная репликация** — только определённые таблицы/столбцы/строки

### Мониторинг
```sql
-- Подписки
SELECT * FROM pg_subscription;
SELECT * FROM pg_stat_subscription;

-- Публикации
SELECT * FROM pg_publication;
SELECT * FROM pg_publication_tables;
```

### Разрешение конфликтов
- Конфликты при двунаправленной репликации
- Ошибки применения логируются
- `disable_on_error = on` — отключить подписку при ошибке

### Целостность ссылок
- При логической репликации используется **REPLICA IDENTITY** для идентификации строк
- Без первичного ключа или replica identity — UPDATE/DELETE невозможны
