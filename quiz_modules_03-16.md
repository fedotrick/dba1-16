## Модуль 03: Конфигурация

### Вопрос 1
**Какой файл конфигурации записывается утилитой ALTER SYSTEM?**

A) postgresql.conf
B) postgresql.auto.conf
C) pg_hba.conf
D) postgresql.override.conf

<details><summary><b>Ответ: B</b></summary>
ALTER SYSTEM записывает параметры в postgresql.auto.conf в каталоге PGDATA. Этот файл считывается после postgresql.conf, поэтому его параметры имеют приоритет.
</details>

### Вопрос 2
**Какой контекст параметра требует перезапуска сервера для вступления изменений в силу?**

A) sighup
B) user
C) postmaster
D) backend

<details><summary><b>Ответ: C</b></summary>
Параметры с context = 'postmaster' требуют перезапуска сервера. Параметры sighup — достаточно reload.
</details>

### Вопрос 3
**Как перечитать файлы конфигурации без перезапуска сервера из SQL?**

A) `SELECT pg_reload_conf();`
B) `SELECT pg_reset_conf();`
C) `SELECT pg_read_conf();`
D) `SELECT pg_apply_conf();`

<details><summary><b>Ответ: A</b></summary>
Функция pg_reload_conf() эквивалентна pg_ctl reload и заставляет сервер перечитать конфигурационные файлы.
</details>

### Вопрос 4
**Если один параметр указан несколько раз в postgresql.conf, какое значение будет использовано?**

A) Первое
B) Последнее
C) Среднее
D) Сервер не запустится

<details><summary><b>Ответ: B</b></summary>
PostgreSQL считывает файл сверху вниз, и при повторном указании параметра используется последнее значение.
</details>

### Вопрос 5
**Как установить параметр work_mem для конкретного сеанса?**

A) `ALTER SYSTEM SET work_mem = '64MB';`
B) `SET work_mem = '64MB';`
C) `ALTER ROLE CURRENT_USER SET work_mem = '64MB';`
D) Изменить postgresql.conf

<details><summary><b>Ответ: B</b></summary>
SET устанавливает параметр только для текущего сеанса. ALTER SYSTEM — для всего экземпляра. ALTER ROLE SET — для сеансов конкретного пользователя.
</details>

## Модуль 04: Общее устройство

### Вопрос 6
**Какой процесс PostgreSQL создаёт новый backend-процесс для каждого клиента?**

A) bgwriter
B) checkpointer
C) postmaster
D) walwriter

<details><summary><b>Ответ: C</b></summary>
Postmaster — главный процесс, слушает входящие соединения и создаёт (fork) backend-процесс для каждого нового клиента.
</details>

### Вопрос 7
**Что такое SAVEPOINT?**

A) Точка восстановления после сбоя сервера
B) Точка внутри транзакции, к которой можно откатиться
C) Контрольная точка WAL
D) Точка создания снимка данных

<details><summary><b>Ответ: B</b></summary>
SAVEPOINT создаёт точку внутри транзакции. ROLLBACK TO sp отменяет изменения от точки сохранения до текущего момента, но не завершает транзакцию.
</details>

### Вопрос 8
**Какие шаги проходит запрос перед выполнением?**

A) Компиляция → Оптимизация → Выполнение
B) Разбор → Переписывание → Планирование → Выполнение
C) Парсинг → Валидация → Выполнение
D) Разбор → Выполнение

<details><summary><b>Ответ: B</b></summary>
Запрос проходит: разбор (parser) → переписывание (rewriter) → планирование (planner) → выполнение (executor).
</details>

### Вопрос 9
**Что даёт использование подготовленных операторов (PREPARE)?**

A) Автоматическую фиксацию транзакций
B) Разбор запроса один раз при повторных вызовах
C) Защиту от потери данных при сбое
D) Автоматическое создание индексов

<details><summary><b>Ответ: B</b></summary>
PREPARE выполняет разбор и переписывание один раз. При последующих EXECUTE эти шаги пропускаются.
</details>

### Вопрос 10
**Что происходит с курсором при завершении транзакции?**

A) Курсор сохраняется
B) Курсор автоматически закрывается
C) Курсор переходит в другую транзакцию
D) Курсор блокируется

<details><summary><b>Ответ: B</b></summary>
Курсоры автоматически закрываются по завершении транзакции. Исключение — курсоры с WITH HOLD.
</details>

### Вопрос 11
**В какой памяти хранятся разобранные запросы и состояние курсоров?**

A) В общей памяти
B) В локальной памяти backend-процесса
C) В WAL
D) В буферном кеше

<details><summary><b>Ответ: B</b></summary>
Разобранные запросы, планы, состояние курсоров, кеш системного каталога, место для сортировок — всё это в локальной памяти backend-процесса.
</details>

## Модуль 05: MVCC

### Вопрос 12
**Что такое MVCC?**

A) Метод сжатия данных
B) Механизм хранения нескольких версий одной строки
C) Тип индекса
D) Метод резервного копирования

<details><summary><b>Ответ: B</b></summary>
MVCC (Multiversion Concurrency Control) — механизм многоверсионности, позволяющий хранить несколько версий одной строки.
</details>

### Вопрос 13
**Какой уровень изоляции используется по умолчанию в PostgreSQL?**

A) Read Uncommitted
B) Read Committed
C) Repeatable Read
D) Serializable

<details><summary><b>Ответ: B</b></summary>
Read Committed — уровень по умолчанию. Снимок строится при начале каждого оператора.
</details>

### Вопрос 14
**Чем отличается Read Committed от Repeatable Read?**

A) Read Committed быстрее
B) В Read Committed снимок строится для каждого оператора, в Repeatable Read — для всей транзакции
C) Repeatable Read допускает грязные чтения
D) Нет отличий

<details><summary><b>Ответ: B</b></summary>
В Read Committed снимок обновляется при каждом операторе. В Repeatable Read снимок создаётся один раз при первом операторе транзакции.
</details>

### Вопрос 15
**Что хранится в файлах pg_xact/?**

A) Данные таблиц
B) Статусы транзакций (clog)
C) Журнал WAL
D) Файлы конфигурации

<details><summary><b>Ответ: B</b></summary>
pg_xact/ содержит файлы статусов транзакций (commit log, clog) — по 2 бита на каждую транзакцию.
</details>

### Вопрос 16
**Одинаково ли по скорости выполняются COMMIT и ROLLBACK?**

A) COMMIT быстрее
B) ROLLBACK быстрее
C) Одинаково — в обоих случаях только устанавливаются биты в clog
D) Зависит от размера транзакции

<details><summary><b>Ответ: C</b></summary>
И фиксация, и обрыв одинаково быстры — достаточно установить соответствующие биты в clog. Данные НЕ откатываются физически.
</details>

### Вопрос 17
**Блокирует ли SELECT другие транзакции?**

A) Да, все пишущие транзакции
B) Да, только те, что меняют те же строки
C) Нет, чтение никогда не блокирует
D) Только при уровне Serializable

<details><summary><b>Ответ: C</b></summary>
При MVCC чтение никогда не блокирует ни читающие, ни пишущие транзакции.
</details>

## Модуль 06: VACUUM

### Вопрос 18
**Какие задачи выполняет VACUUM?**

A) Только удаляет мёртвые строки
B) Удаляет мёртвые строки, обновляет карты видимости и свободного пространства, собирает статистику, замораживает строки
C) Только собирает статистику
D) Только замораживает строки

<details><summary><b>Ответ: B</b></summary>
VACUUM выполняет несколько задач: вычищает мёртвые версии, обновляет карту видимости и свободного пространства, собирает статистику, замораживает старые версии.
</details>

### Вопрос 19
**Уменьшает ли обычный VACUUM размер файла таблицы?**

A) Да, всегда
B) Да, если много мёртвых строк
C) Нет, место не возвращается ОС
D) Только с ключом FULL

<details><summary><b>Ответ: C</b></summary>
Обычный VACUUM освобождает место внутри страниц, но не возвращает его операционной системе. Только VACUUM FULL уменьшает файл.
</details>

### Вопрос 20
**Для чего нужна карта видимости (visibility map)?**

A) Для шифрования данных
B) Для оптимизации очистки и ускорения Index Only Scan
C) Для репликации
D) Для резервного копирования

<details><summary><b>Ответ: B</b></summary>
Карта видимости отмечает страницы, где все строки видны во всех снимках. Это позволяет очистке пропускать такие страницы и ускоряет Index Only Scan.
</details>

### Вопрос 21
**Что делает параметр autovacuum_vacuum_scale_factor?**

A) Определяет минимальное количество обновлений для запуска VACUUM
B) Определяет долю от размера таблицы, при превышении которой запускается VACUUM
C) Определяет частоту запуска автоочистки
D) Определяет количество рабочих процессов

<details><summary><b>Ответ: B</b></summary>
autovacuum_vacuum_scale_factor (по умолчанию 0.2 = 20%) — доля от размера таблицы. Автоочистка запускается когда количество обновлений превышает threshold + scale_factor * размер.
</details>

### Вопрос 22
**Чем REINDEX CONCURRENTLY отличается от обычного REINDEX?**

A) Работает быстрее
B) Не блокирует запись в таблицу
C) Работает для системных индексов
D) Может выполняться внутри транзакции

<details><summary><b>Ответ: B</b></summary>
REINDEX CONCURRENTLY не блокирует запись в таблицу. Но он дольше, не транзакционный, и не работает для системных индексов и индексов EXCLUDE.
</details>

## Модуль 07: Буферный кеш и WAL

### Вопрос 23
**Какой алгоритм вытеснения используется в буферном кеше?**

A) FIFO
B) LRU (Least Recently Used)
C) Random
D) LFU (Least Frequently Used)

<details><summary><b>Ответ: B</b></summary>
LRU — вытесняется страница, которая дольше всего не использовалась. Если буфер «грязный», страница сначала записывается на диск.
</details>

### Вопрос 24
**Что такое LSN?**

A) Имя сервера
B) Log Sequence Number — 64-битное смещение записи WAL
C) Тип блокировки
D) Имя файла конфигурации

<details><summary><b>Ответ: B</b></summary>
LSN (Log Sequence Number) — 64-разрядное число, смещение записи в байтах от начала журнала. Записывается как два 32-битных числа: 0/2378D28.
</details>

### Вопрос 25
**Для чего нужна контрольная точка (checkpoint)?**

A) Для создания резервной копии
B) Для сброса всех грязных буферов на диск и ограничения размера WAL
C) Для переключения на реплику
D) Для создания новой базы данных

<details><summary><b>Ответ: B</b></summary>
Checkpoint сбрасывает все грязные буферы на диск, гарантируя сохранность изменений и ограничивая объём WAL, необходимого для восстановления.
</details>

### Вопрос 26
**Какой процесс выполняет асинхронную запись WAL?**

A) checkpointer
B) bgwriter
C) walwriter
D) autovacuum launcher

<details><summary><b>Ответ: C</b></summary>
walwriter — фоновый процесс, асинхронно записывающий WAL на диск.
</details>

### Вопрос 27
**Какой уровень wal_level необходим для логической репликации?**

A) minimal
B) replica
C) logical
D) full

<details><summary><b>Ответ: C</b></summary>
Для логической репликации требуется wal_level = logical. Для физической достаточно replica.
</details>

### Вопрос 28
**Что произойдёт при останове сервера в режиме immediate?**

A) Сервер выполнит контрольную точку и остановится
B) Сервер подождёт отключения всех клиентов
C) Сервер остановится мгновенно без checkpoint, при запуске потребуется восстановление через WAL
D) Сервер откатит все транзакции и остановится

<details><summary><b>Ответ: C</b></summary>
Режим immediate — мгновенная остановка без checkpoint. При следующем запуске PostgreSQL войдёт в режим восстановления и проигрывает WAL.
</details>

### Вопрос 29
**Что НЕ защищает WAL?**

A) Страницы таблиц
B) Страницы индексов
C) Временные и UNLOGGED таблицы
D) Статус транзакций (clog)

<details><summary><b>Ответ: C</b></summary>
WAL не защищает временные (TEMP) и нежурналируемые (UNLOGGED) таблицы — их данные не записываются в журнал.
</details>

**Что такое search_path?**

## Модуль 11: Низкий уровень

A) 256 МБ
B) 512 МБ
C) 1 ГБ
D) 2 ГБ

<details><summary><b>Ответ: C</b></summary>

### Вопрос 45
**Что такое TOAST?**

A) Метод индексации
B) Механизм хранения длинных значений за пределами основной таблицы
C) Тип репликации
D) Метод резервного копирования

<details><summary><b>Ответ: B</b></summary>
TOAST (The Oversized-Attribute Storage Technique) — механизм для хранения значений, не помещающихся в страницу. Значение разбивается на chunks ~2 КБ в отдельной таблице.
</details>

### Вопрос 46
**Какая стратегия хранения сжимает данные перед выносом?**

A) plain
B) extended
C) external
D) main

<details><summary><b>Ответ: B</b></summary>
extended — сначала сжимает, если всё равно не влезает — выносит. external — не сжимает, сразу выносит. main — сжимает, выносит только если необходимо.

### Вопрос 47

pg_total_relation_size() = данные + TOAST + все индексы. pg_table_size() = данные + TOAST + FSM + VM (без дополнительных индексов).
</details>

## Модуль 15: Физическая репликация

### Вопрос 67
**Какой процесс на основном сервере отправляет WAL реплике?**

A) walreceiver
B) walsender
C) walwriter
D) startup

<details><summary><b>Ответ: B</b></summary>
walsender (wal sender) — на основном, отправляет WAL. walreceiver — на реплике, принимает. startup — на реплике, проигрывает.
</details>

### Вопрос 68
**Какой тип репликации используется по умолчанию?**

A) Синхронная
B) Асинхронная
C) Полусинхронная
D) Логическая

<details><summary><b>Ответ: B</b></summary>
Асинхронная репликация — по умолчанию. Основной сервер не ждёт подтверждения от реплики.
</details>

### Вопрос 69
**Для чего нужны слоты репликации?**

A) Для ограничения скорости репликации
B) Для гарантии, что WAL не будет удалён до получения репликой
C) Для аутентификации реплики
D) Для шифрования WAL

<details><summary><b>Ответ: B</b></summary>
Слоты репликации гарантируют, что WAL-файлы не будут удалены на основном сервере, пока все реплики их не получат.
</details>

### Вопрос 70
**Какое представление на основном сервере показывает статус репликации?**

A) pg_stat_activity
B) pg_stat_replication
C) pg_replication_slots
D) pg_stat_wal_receiver

<details><summary><b>Ответ: B</b></summary>
pg_stat_replication показывает: состояние, sent_lsn, write_lsn, flush_lsn, replay_lsn для каждой подключённой реплики.
</details>

### Вопрос 71
**Что делает параметр hot_standby_feedback = on на реплике?**

A) Включает режим чтения на реплике
B) Реплика сообщает основному о своих снимках, предотвращая конфликты с VACUUM
C) Ускоряет репликацию
D) Включает синхронную репликацию

<details><summary><b>Ответ: B</b></summary>
hot_standby_feedback = on позволяет реплике информировать основной сервер о своих активных снимках, чтобы VACUUM не удалял нужные реплике версии строк.
</details>

### Вопрос 72
**Какая функция повышает реплику до основного сервера?**

A) pg_switch_wal()
B) pg_promote()
C) pg_standby()
D) pg_failover()

<details><summary><b>Ответ: B</b></summary>
pg_promote() — SQL-функция для повышения реплики. Также: pg_ctl promote.
</details>

### Вопрос 73
**Как настроить задержанную репликацию на 5 минут?**

### Вопрос 75
**Что такое публикация в логической репликации?**

B) Набор таблиц, изменения которых отправляются подписчикам

D) Слот репликации

<details><summary><b>Ответ: B</b></summary>
Публикация — набор таблиц (и опционально столбцов/строк), изменения которых отправляются подписчикам.
</details>

### Вопрос 76
**Реплицируются ли DDL-команды при логической репликации?**

A) Да, все
B) Только CREATE TABLE
C) Нет, не реплицируются

<details><summary><b>Ответ: C</b></summary>
DDL-команды НЕ реплицируются при логической репликации. Структуру таблиц нужно создавать на подписчике вручную.
</details>

### Вопрос 77
**Что такое REPLICA IDENTITY?**

A) Копия таблицы на реплике
B) Механизм идентификации строк для UPDATE/DELETE при репликации
C) Тип индекса

<details><summary><b>Ответ: B</b></summary>
REPLICA IDENTITY определяет, какие столбцы используются для идентификации строк при UPDATE и DELETE на подписчике. По умолчанию — первичный ключ.
</details>

### Вопрос 78
**Какое значение REPLICA IDENTITY включает все столбцы?**

A) DEFAULT
B) FULL
C) ALL
D) COMPLETE

<details><summary><b>Ответ: B</b></summary>
REPLICA IDENTITY FULL — все столбцы используются для идентификации строк. DEFAULT — только первичный ключ. NOTHING — нет идентификатора.
</details>

### Вопрос 79
**Можно ли реплицировать только определённые столбцы таблицы?**

A) Нет, только всю таблицу
B) Да, через публикацию с указанием столбцов

Начиная с PG15, публикация может включать только определённые столбцы: `CREATE PUBLICATION FOR TABLE t (col1, col2);`
</details>

### Вопрос 80
**Какой процесс на подписчике применяет изменения?**

A) walsender
B) logical replication worker

logical replication worker (на подписчике) получает изменения от издателя и применяет их.
